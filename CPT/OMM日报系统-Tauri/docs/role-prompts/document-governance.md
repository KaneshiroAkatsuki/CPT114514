# 文档治理规则

更新时间：2026-07-04 02:20 +08:00
适用范围：杰、寇、凯，以及后续 Codex/opencode 窗口。

## 目标

让三角色协作时能快速接手项目，又不会因为文档过长、重复、失真而浪费上下文。

## 文档分层

1. 根入口：`AGENTS.md`
   - 跨 Codex/opencode 的最小入口。
   - 只放项目当前规则、硬约束、常用命令和阅读路线。

2. 当前状态：`CURRENT_STATUS.md`
   - 只记录稳定状态、最新版本、重要限制和最新交付。
   - 如果发现它滞后，先在交接中标注，再由杰统一更新。

3. 角色提示词：`docs/role-prompts/*.md`
   - 只写角色职责、边界、启动提示词和输出要求。
   - 不写功能历史、测试流水账或大段实现细节。

4. 当前交接：`docs/role-prompts/handoffs/*.md`
   - 只写接手当前任务必须知道的内容。
   - 一个主题一个文件，优先短交接。

5. 归档：`docs/archive/`、`docs/role-prompts/handoffs/archive/`
   - 只为追溯历史存在。
   - 非必要不读取，不作为新窗口默认上下文。

6. 轻量工作记录：`docs/role-prompts/activity-log.md`
   - 只记录三角色直接处理的低风险小事项。
   - 一条一到两行，避免变成聊天记录仓库。

## 命名规则

交接文件统一命名：

```text
YYYY-MM-DD_HHMM_vX.Y.Z_to-角色_主题.md
```

示例：

```text
2026-07-04_0205_v5.8.1_to-jay_rehandoff-maintenance.md
```

研究记录如需保留，使用：

```text
YYYY-MM-DD_HHMM_research_主题.md
```

## 长度控制

- `AGENTS.md` 保持短入口，目标是不超过 32 KiB。
- 单个交接文件目标不超过 120 行。
- 单个角色提示词目标不超过 180 行。
- 如果文档继续变长，优先新建续接文件或归档旧文件，不继续堆叠。

## 更新触发

需要更新文档的场景：

- 版本发布、便携包更新、重要路径改变。
- 功能边界、数据库结构、清理规则、登录权限或打包策略改变。
- 上下文即将交接给新窗口或新角色。
- 外部工具、插件、Skill、MCP 被长期引入。
- 用户明确说“记下来”“下次默认”“以后按这个”。
- 三角色在职责范围内直接处理了低风险小事项，且其他窗口需要知道。

不需要写入长期文档的内容：

- 临时讨论、未定方案、失败尝试流水账。
- 大段终端输出、大段源码、大量截图说明。
- 可通过 `git status`、源码或 release manifest 直接查到的重复信息。
- 私密信息、个人数据明细。用户明确提供给本地验收使用的测试账号信息，只能放在必要的角色验收文档或短交接中，不能扩散到日志、提交信息、公开说明或普通 UI 文案。

## 交接流程

1. 交接人先写短摘要：当前目标、已完成、未完成、风险、下一步。
2. 明确“建议读取范围”，不要让接手者读整个 docs。
3. 旧交接被新交接替代后，移动到 `handoffs/archive/`。
4. 接手者先按建议读取范围工作，发现缺口再补读。
5. 稳定结论由杰写入 `CURRENT_STATUS.md` 或根入口；寇和凯不随意扩写长期文档。

## 小事自治与记录

三角色可以在各自职责范围内直接处理低风险小事项，不必每次询问用户或杰。

可以直接处理的例子：

- 明确错字、版本号显示、入口路径、帮助文案中的小错误。
- 与当前任务直接相关的轻量 UI 一致性修正。
- 明确 bug 的小范围修复和基础自测。
- 验收记录、复现步骤、测试清单的整理。
- 不影响用户数据和核心逻辑的构建脚本或文档指针修正。

不能当作小事项直接处理的例子：

- 数据库结构迁移、默认规则覆盖、清空或重建数据。
- 真实清理用户文件、日期文件夹、备份或旧发布包。
- 登录权限、发布策略、版本号升级、完整打包发布。
- 核心识别算法、耗时推荐算法、日报生成规则和清理策略的大改。
- 大范围 UI 重构、模块定位变化或跨模块行为改变。

完成小事项后必须写短记录：

- 优先写入 `docs/role-prompts/activity-log.md`。
- 如果小事项属于某个正在交接的任务，也可以写在当前交接文件的“已处理”小节。
- 记录格式建议：时间、角色、事项、主要文件、验证结果。
- 如果事项变成稳定规则或发布状态，由杰决定是否同步到 `CURRENT_STATUS.md`。

## 网页与新技术

涉及新技术、插件、Skill、MCP、Codex/opencode 行为、文档管理经验或不确定事实时，要主动访问网页参考。

网页参考按可信度排序：

1. 官方文档和项目维护者文档。
2. 维护活跃、issue/commit 清晰的 GitHub 仓库。
3. 被多个独立来源印证的技术文章。
4. 社交媒体、论坛和视频只作为线索，不作为定论。

采用外部建议前必须做三件事：

- 记录来源链接和查看日期。
- 判断它解决的是当前项目真实问题，还是只适合别的团队。
- 能本地验证的必须本地验证，不能验证的要标明不确定。

## 已参考的成熟实践

- `AGENTS.md` 作为 AI 代理入口，适合 Codex 与 opencode 共享。
- Memory Bank 类实践适合长期状态，但不能变成聊天记录仓库。
- Codex Skills 适合沉淀可复用流程，MCP 适合连接外部数据和工具。
- 外部 MCP/Skill/插件必须先做来源、安全和维护性判断，再决定是否长期引入。
