# 角色协作文件夹

本目录只存放三角色协作所需的轻量提示词和交接文件，避免把上下文塞进主项目文档。

## 文件结构

- `jay.md`：杰，项目总控、架构设计、主要文档和发布决策。
- `kou.md`：寇，代码实现、简单检查和简单修复。
- `kai.md`：凯，验收测试、打包测试和质量反馈。
- `core-memory.md`：上下文压缩或换窗口后必须恢复的项目核心规则。
- `document-governance.md`：文档分层、交接命名、归档和网页参考规则。
- `tooling-policy.md`：工具、插件、Skill 与 MCP 的使用规则。
- `activity-log.md`：三角色直接处理低风险小事项后的短记录。
- `handoffs/`：角色之间的短交接文件。
- `handoffs/archive/`：旧交接归档，非必要不读取。
- `../archive/legacy-root/`：从项目根目录移走的旧版根文档。
- `../archive/legacy-current-layer/`：从 `docs/` 当前层移走的旧接力链和阶段记录。

## 交接命名

新交接文件统一使用：

```text
YYYY-MM-DD_HHMM_vX.Y.Z_to-角色_主题.md
```

示例：

```text
2026-07-04_0140_v5.8.1_to-jay_context-handoff.md
```

## 文档治理规则

1. 交接文件只写当前窗口需要知道的内容，不复述完整历史。
2. 单个交接文件过长时，创建新的续接文件，旧文件移入 `handoffs/archive/`。
3. 归档文件非必要不读取，只在追溯历史、查错或杰明确要求时读取。
4. 稳定结论再写入长期文档，临时想法只保留在交接文件或任务说明中。
5. 三个角色首轮互动时，需要互相提醒：不要乱读文档、不要乱建文档、交接文件要命名清楚并及时归档。
6. 三个角色要主动合理使用工具、插件、Skill 和 MCP；缺少必要能力时可寻找可信来源安装，但长期工具由杰确认。
7. 涉及新技术、插件、Skill、MCP、Codex/opencode 行为或不确定事实时，要主动访问网页参考；网页不能直接当真，必须结合本地验证。
8. 低风险小事项可以由各角色在职责范围内直接处理，不必每次问用户；完成后写入 `activity-log.md` 或当前交接，避免其他模型不知道。
