# 工具、插件、Skill、MCP 与网页参考规则

## 总原则

三角色应主动、合理地使用 Codex 可用的工具、插件、Skill、MCP 和网页参考，不要纯靠记忆或手工重复劳动。

工具使用要服务于当前任务，不能为了“显得专业”而过度调用。

## 优先级

1. 先使用当前环境已经提供的工具、Skill、插件和 MCP。
2. 需要发现延迟加载工具时，优先使用 Codex 的工具发现能力。
3. 缺少必要能力时，可以查找可信来源，例如官方文档、官方插件、项目维护者仓库或 GitHub 上明确可信的开源实现。
4. 涉及新技术、插件、Skill、MCP、opencode/Codex 行为、外部标准或不确定事实时，要主动访问网页参考。
5. 安装新插件、Skill 或 MCP 前，要说明用途、来源、版本、维护状态、风险和是否会影响项目环境。
6. 影响范围不明或需要长期保留的工具，由“杰”确认后再安装。

## 网页参考

网页资料只能作为证据来源，不能直接当成项目事实。

可信度从高到低：

1. 官方文档、官方 registry、项目维护者仓库。
2. 维护活跃、issue/commit 清晰、许可证明确的 GitHub 项目。
3. 多个独立来源相互印证的技术文章。
4. 社交媒体、视频、论坛和博客评论只作为线索。

采用网页建议前要做：

- 记录来源链接和查看日期，避免下次不知道依据来自哪里。
- 用本地源码、命令、测试或小实验验证。
- 如果无法验证，明确标注为“参考建议”或“不确定”。

## 角色边界

- 杰：决定是否引入新的插件、Skill 或 MCP；维护工具使用规则；控制安全和上下文成本。
- 寇：为实现和自测主动使用合适工具；缺少工具时先向杰说明需求，不擅自引入长期依赖。
- 凯：为验收主动使用浏览器、截图、UI 自动化、文件检查等工具；缺少测试工具时向杰反馈。

## 适合主动使用的场景

- 需要实际登录、截图、浏览器操作、UI 验证时，优先使用浏览器或计算机控制类工具。
- 需要读写 Office、PDF、表格、图片、报告时，优先使用对应 Skill。
- 需要 GitHub、Issue、PR、CI 信息时，优先使用 GitHub 插件或 MCP。
- 需要结构化查询、数据库、日志、资源列表时，优先使用 MCP。
- 需要重复构建、打包、清理、检查时，优先写成脚本或使用已有脚本。
- 需要判断新工具或新方法时，先查官方文档和维护活跃的 GitHub，再决定是否试用。

## 禁止事项

- 不安装来源不明的插件、Skill 或 MCP。
- 不为了小问题引入重型工具。
- 不把工具安装记录塞进大型历史文档；需要记录时放入短交接或由杰写入稳定状态文档。
- 不让工具自动修改用户数据、日期文件夹、备份文件或数据库，除非任务明确要求且已确认安全边界。
- 不把网页里的做法未经验证直接写成项目规则。

## 首轮提醒

杰、寇、凯第一次互相协作时，需要提醒对方：

- 主动使用合适的工具、Skill、插件和 MCP。
- 缺少工具时可以寻找可信来源安装，但要先说明必要性和风险。
- 新增长期工具由杰确认和记录。
- 新技术和外部经验要主动查网页，但要带来源、验证和不确定性说明。
