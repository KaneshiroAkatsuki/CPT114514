## 1. 本轮用户确认的新事实

用户提供了最新模板文件夹：

```text
C:\Users\Administrator\Desktop\勿动\识别文件夹
```

识别时只看非 `OLD` 文件夹/文件。本目录里的文件夹代表工站名称，但存在阶段目录与特殊命名，需要规则化。

核心事实：

1. **品名通常是三位数字**，不是完整料号。
2. **烧结盘允许特殊品名 `0.2`、`0.25`**。
3. `EVT-SLIDER`、`EVT ALT-2`、`X168` 这类是机器/项目/阶段信息，不是平时测的料的品名。
4. `806-65036-04` 的品名是 `036`；`806-65037` 的品名是 `037`。
5. 品名会在多个工站重复，所以识别结果必须同时保留 **工站 + 品名 + 测试类型**，不能只靠品名判断。
6. `FAI` 代表开发；如果没有其他明确工站，或遇到奇怪工站名，可用 `开发` 兜底。
7. 文件名出现 `首件` 填 `首件`，出现 `制程` 填 `制程`，否则填 `尺寸` 或 `测试尺寸`。
8. CNC 工站固定品名是 `035`；整形工站里的 CNC 也按 `035`。
9. 焊接很特殊，用户没怎么测过，当前常见品名大概是 `289`、`290`、`424`-`429`。

## 3. 不能删除的既有能力

本轮增强不能删除现有识别能力，只能加强：

1. 普通任务文件夹原有解析逻辑继续保留。
2. 手量候选自动发现继续保留。
3. `-手量-姓名` 继续识别为测量员。
4. `姓名送测` 要识别为送测人，例如 `安容克送测 -> 安容克`。
5. `CMM-姓名` / `OMM-姓名` 后面的人名不能当品名。
6. 数量如 `96PCS` 继续识别。
7. 手量耗时不依赖文件夹名；以人工输入为主。输入 `2` 默认按 2 小时，不是 2 分钟。

## 4. 建议新增：独立“识别补充”窗口

用户明确希望未来可能出现新料号/新工站时，可以在独立窗口补充，不必每次改代码。

### 4.1 窗口名称

建议组件名：

```text
RecognitionRulesDialog.tsx
```

界面入口建议放在“设置/配置”区域，按钮名：

```text
识别补充
```

### 4.2 窗口应该包含的内容

1. **内置规则说明**
   - 展示当前内置规则：CNC=035、FAI=开发、806 料号取后三位、烧结盘、焊接等。
   - 内置规则只读，避免用户误删基础能力。

2. **自定义工站别名**
   - 字段：别名、归一工站名、默认类型。
   - 示例：`FAI(全尺寸） -> 开发 -> 尺寸`。

3. **自定义品名规则/别名**
   - 字段：匹配关键词/正则、输出品名、适用工站、备注。
   - 示例：`806-65040 -> 040`，`X661-烧结盘 -> 661`。

4. **忽略词**
   - 如 `EVT`、`ALT-1`、`ALT-2`、`X168`、`BIN2`、`BIN4`、`AOI`、`FAI9`、日期等。
   - 避免这些词被误当品名。

5. **测试识别**
   - 输入或粘贴一个文件夹名/文件名。
   - 显示解析结果：工站、品名、测试类型、送测人、数量、测量员、置信度/来源。
   - 显示“命中的规则”，方便用户知道为什么这么识别。

6. **导入模板文件夹（可后置）**
   - 让用户选择一个模板根目录，扫描非 OLD 文件，生成候选规则。
   - 第一版可以不实现自动导入，只保留未来入口。

### 4.3 持久化：补充规则必须独立成文件

用户补充的识别规则不要直接塞进 `config.json`。原因：

1. `config.json` 未来可能被“恢复默认设置”、迁移、误清空或重新生成。
2. 识别补充规则是用户长期积累的知识库，价值比普通界面配置更高。
3. 如果规则跟配置放在同一个 JSON 内，用户更新/重置配置时容易误删补充内容。

建议采用双文件策略：

```text
config.json                  # 普通界面配置，可随设置修改/重置
recognition-rules.json       # 用户识别补充规则，独立保存
```

两个文件放在同一个配置目录里：

```text
AppData 模式：
%APPDATA%\OMM日报系统\config.json
%APPDATA%\OMM日报系统\recognition-rules.json

便携版模式：
<当前识别到的便携版配置目录>\config.json
<当前识别到的便携版配置目录>\recognition-rules.json
```

`config.json` 里最多只保存规则文件名/路径，不保存完整规则内容：

```ts
recognition_rules_path?: string; // 默认 recognition-rules.json
```

真正的 `recognition-rules.json` 建议结构：

```ts
{
  version: 1;
  updated_at?: string;
  station_aliases?: Array<{
    alias: string;
    station: string;
    default_test_type?: '首件' | '制程' | '尺寸' | '测试尺寸';
  }>;
  product_aliases?: Array<{
    pattern: string;
    product: string;
    station?: string;
    note?: string;
  }>;
  ignored_tokens?: string[];
  welding_rules?: Array<{
    pattern: string;
    product: string;
    note?: string;
  }>;
  sinter_plate_rules?: Array<{
    pattern: string;
    products: string[];
    note?: string;
  }>;
}
```

删除/清空规则的原则：

```text
普通“恢复默认配置”不删除 recognition-rules.json；
只有用户在“识别补充”窗口里点“清空全部补充规则”或删除某条规则，才修改 recognition-rules.json；
用户手动删除 recognition-rules.json 时，程序应自动回到只有内置规则的状态，并提示“未找到识别补充文件，可重新创建”。
```

内置规则与自定义规则的关系：

```text
内置规则永远存在；
自定义规则用于补充/覆盖；
不要用自定义规则替换整个内置规则表。
```

帮助页必须同步说明：

1. 识别规则有“内置规则”和“补充规则”两层。
2. 补充规则文件名是 `recognition-rules.json`。
3. 它和 `config.json` 在同一个配置目录。
4. 重置普通配置不会清空补充规则。
5. 如需删除补充规则，请在“识别补充”窗口中删除单条或清空全部。

## 5. 建议代码结构

### 5.1 新增识别模块

建议新增：

```text
src/lib/recognitionRules.ts
```

职责：

1. 保存内置规则常量。
2. 提供纯函数：

```ts
recognizeStation(name: string, context?: RecognitionContext): RecognitionResult
recognizeProductCodes(name: string, station?: string): string[]
recognizeTestType(name: string): '首件' | '制程' | '尺寸'
recognizeManualTaskFromFolderV2(name: string, options?: RecognitionOptions): Partial<RealManualTask>
```

3. 返回结果应带来源：

```ts
{
  station?: string;
  products?: string[];
  product?: string;
  test_type?: string;
  sender?: string;
  quantity?: string;
  operator?: string;
  matched_rules?: string[];
  warnings?: string[];
}
```

### 5.2 保持兼容

现有 `src/lib/utils.ts` 中的 `recognizeManualTaskFromFolder()` 不要直接删除。建议改成调用新模块：

```ts
export function recognizeManualTaskFromFolder(folderName: string): Partial<RealManualTask> {
  return recognizeManualTaskFromFolderV2(folderName);
}
```

这样旧调用点不需要大面积改。

### 5.3 类型补充

`RealManualTask` 建议新增：

```ts
station?: string;
matched_rules?: string[];
recognition_warnings?: string[];
```

手量弹窗显示字段改为：

```text
工站
品名
送测人
测试日期
数量
耗时
测量员
```

不需要显示：


### 5.4 手量耗时输入

手量耗时以人工输入为主：

```text
2      -> 120 分钟
2.5    -> 150 分钟
3h/3H  -> 180 分钟
150分钟/150分/150m -> 150 分钟
```

界面提示：

```text
手量耗时默认按小时输入；如需按分钟，请写“90分钟”或“90m”。
```

未填写耗时不应作为“识别失败”强警告，只在生成前阻止并提示“请填写手量耗时”。

## 6. 识别优先级建议

### 6.1 工站优先级

1. 路径/上级文件夹明确工站。
2. 文件名包含明确工站词：`CNC`、`射出`、`整形`、`烧结`、`焊接`、`电镀`、`镭雕`、`二维`、`FAI`。
3. `FAI` 兜底为 `开发`。
4. 无法判断且文件名奇怪时，按 `开发` 兜底，但要给 warning。

### 6.2 品名优先级

1. CNC 或整形+CNC 固定 `035`。
2. 烧结盘特殊规则：小数品名、多品名拆分。
3. 焊接特殊规则：`289/290/41424-41429`。
4. `806-xxxxx-xx` / `806-xxxxx` 取 `xxxxx` 后三位。
5. 文件名开头三位数字可作为品名，但要避开日期、数量、工单、人员、`CMM/OMM/手量` 后姓名。
6. 自定义规则补充/覆盖。

### 6.3 测试类型优先级

```text
包含 首件 -> 首件
包含 制程 -> 制程
否则 -> 尺寸/测试尺寸
```
