这份文档记录用户习惯、协作方式、项目安全边界和模型使用约定。

## 用户偏好

- 用户希望方向清楚后直接实现，不喜欢只停留在建议。
- 大改 UI、架构、核心流程前要先设计和讨论。
- 用户很重视界面质感，功能可用只是底线。
- 当前 UI 方向是 Apple-inspired：干净、克制、柔和、现代，但不要花哨。
- 用户不喜欢拥挤页面、不明所以的技术名词和堆叠数据。
- 帮助文档必须跟随功能和界面同步更新。
- 用户要求打包或发布时，版本号要同步更新。
- 一般只需要便携包，不需要专门交付安装包。

## 模型和 token 使用习惯

用户已经开通 Pro，但仍然关注 token 消耗。

建议按这个规则判断：

- 小 bug、普通代码、局部 UI 调整：Codex 直接写。
- 大量重复代码或低风险搬运：可以建议交给 op/Kimi 节省额度。
- 复杂但范围不大的设计和代码：先让用户切到“高”，先设计再写。
- 关键模块设计、架构重组、清理模块安全边界、排程核心等：建议用户切到“超高”，先方案后执行。

Codex 无法替用户切换模型或推理强度，需要主动提醒。

## 安全和隐私习惯

- 不要在普通 UI 文案中过度暴露用户私人浏览器或私人工具细节。
- 原“个人清理”后续应改成更正式的 `数据管理局` 或机台/数据维护语境。
- 管理员功能必须前端限制和后端限制都存在。
- 不要把 PIN、密码写进文档、日志、提交信息或最终回复。
- 不要随便清理 `HTTP_PROXY` / `HTTPS_PROXY`，Codex 或开发环境可能依赖代理。
- 开发清理功能时默认用模拟运行和预览，不要擅自真实清理。
- 涉及进程结束、代理清理、WiFi 切换、回收站清理时，真实执行前要有清单和确认。

## UI 习惯

- 软件要像真正的工具，不要做成营销落地页。
- 不要在应用内做夸张 hero 页面。
- 避免卡片套卡片。
- 文案要短、清楚、用户能懂。
- 图标优先用 lucide。
- 弹窗、设置页、权限提示要保持统一 Apple-inspired 风格。
- 管理员专用功能可以显示入口，但访客点击时要弹出一致的权限提示。
- 帮助和关于软件要使用用户语言，不要堆内部代码名。

## 日报算法待办记忆

- 下次每件时间默认建议记为 `2~5 分钟/件`。
- 普通测料偏多，普通 OMM/CMM 测料最低容许 `1 分钟/件`。
- CNC 任务最低要求 `20 分钟/任务`，不能按普通件数规则压得过低。
- 时间不够时，推荐逻辑应优先倾向补足日报缺口，并说明建议时间的计算依据。
- 手量补录应给推荐耗时，不能只让用户自己猜；如果缺口过大，应提示补录手量/其他事项，而不是强行拉高每件时间。
- 手量项目在补录/生成展示里后面要写 `手量`。
- 送测日期不要写 `6月28日` 这种中文日期；日期本体写 `6.28`，班次字段只写 `A`、`B` 或留空。

## 交流方式

- 语气温和、直接、实用。
- 能自己判断的事就推进，不要频繁把小选择抛回给用户。
- 风险较高时要讲清楚原因，并给更安全的方案。
- 用户有时表达较散，先理解意图，再整理成可执行方案。
- 如果涉及危险清理或数据改动，优先模拟运行、列清单、再确认。
- 三角色的启动和交流提示词统一放在 `docs/role-prompts/`；杰首次对接寇或凯时，要提醒他们先读取各自提示词，也可以让寇转告凯。
